## Fork branch model for cross-machine Open Brain dev (kae36/OB1)

Established a two-lane fork workflow so work continues seamlessly across the **Razer laptop** and the **M7GASG5 desktop**.

### Remotes (on this laptop)
- `fork` → `github.com/kae36/OB1` (personal fork — push everything here)
- `origin` → `github.com/NateBJones-Projects/OB1` (upstream)
- NOTE: a fresh `git clone` of the fork names it `origin` by default; align names per machine.

### Branch lanes
- **`kae36/personal`** — long-lived personal everything-branch; **now the fork's default branch**. Holds the recipe fix + tracked `.memories/` + `OPENROUTER_AUDIT.md`. Never PR'd upstream. Daily work happens here.
- **`contrib/<topic>`** — clean, upstream-bound PR branches. `contrib/kae36/url-batch-import` = PR #339 head (`31b1117`, OPEN, untouched). `contrib/kae36/url-import-fetch-improvements` = `51cc223`, holds ONLY the recipe fix (`import-urls.ts`), ready to feed #339 later via cherry-pick.
- **`main`** — clean upstream mirror; synced `9ff6135 → fa405fd` (+122) and pushed to fork. `origin/main == fork/main == fa405fd`.

### Why
PR #339 is unlikely to merge soon (new contributor, no approver pull), so the user keeps developing in their own fork rather than waiting. Separation is structural: personal files never get `git add`-ed onto contrib branches, so a future #339 update won't drag personal tooling/audit/memory upstream.

### Key decisions
- **`.memories/` committed to fork** (dropped from `.gitignore`) so Goldfish dev memory travels between machines. `.claude`/`.julie`/`.julieignore` stay ignored.
- Clipboard PNG (`2026-05-05 ...png`) left untracked deliberately.
- Recipe fetch fix kept **personal-only for now** (not pushed into PR #339).

### Cross-machine caveat
Repo work + `.memories/` travel via the fork. Claude Code's OWN project memory (`~/.claude/projects/.../memory/*.md`) lives outside the repo and does NOT clone — needs separate per-machine sync if wanted on the desktop.
